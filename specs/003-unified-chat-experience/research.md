# Phase 0 Research: Unified Chat Experience

All items below were unknowns or open design questions in the Technical Context / Constitution Check. Each resolves to a Decision, Rationale, and Alternatives Considered, grounded in the existing codebase (read in full for this feature — see plan.md's Project Structure for the exact files) and the approved design sources (`design/design.md`, `design/motion.md`, `design/screen_map.md`, `design/references/*.png`).

## R1. Conversation-attributed inference ownership (app-wide single-flight, background continuation)

**Decision**: Keep `InferenceActivityLock` and `InferenceQueue` as true process-wide singletons — there is still exactly one inference in flight app-wide at any time, unchanged. Replace `inferenceStore.ts`'s single module-scoped pointers (`activeSessionId: string | null`, `lastSavedSession: QASession | null`, `activeTurn: PendingTurn | null`) with a **conversation-keyed map** (`Map<conversationId, ConversationRuntimeState>`) owned by a new `conversationStore.ts`. Every call to `InferenceQueue.submit()` carries an explicit `conversationId`; the queue's subscription bridge (currently `queue.subscribe(...)` mirroring into a single `InferenceStoreState`) instead looks up and updates only the map entry for the `conversationId` that owns the currently in-flight turn. Screens read their own conversation's slice of the map (`conversationStore.getConversationState(conversationId)`), not a single global `status`/`response` pair. Cancellation (`queue.cancel()`) remains a single app-wide action, but the UI only exposes a cancel control from within the generating conversation's own screen (per spec Edge Cases), so in practice it is always invoked with the correct `conversationId` in scope. Ownership is keyed not just by `conversationId` but by the specific `originatingUserMessageId`/`assistantMessageId` pair the request is answering (data-model.md `ConversationRuntimeState`) — a positional turn index is deliberately not part of this key, since it would be ambiguous across retries and unnecessary once every message has a stable identity (data-model.md's Message Creation Invariant).

**Rationale**: This is the minimal change that satisfies the corrected spec's FR-012/014/015/016 without weakening Constitution Principle II. The single-flight *lock* (only one generation may run) is untouched; only *attribution* (which conversation's UI/state a running or completed generation belongs to) changes from "whatever happens to be the one active thread" to "an explicit, stable key." This directly fixes the spec's core complaint about the current design: `getFollowUpBaseSession`'s continuity check (`lastSavedSession.imagePath !== request.imagePath`) and `AnswerScreen`'s cancel-on-unmount effect both assume there is only ever one conversation "in play" at a time, which cannot represent "conversation A is still generating while the user is looking at conversation B."

**Alternatives considered**:
- *Multiple concurrent `InferenceQueue` instances, one per conversation* — rejected outright: violates Principle II and the spec's explicit non-goal ("multiple simultaneous inference requests"); also would let two image-preprocessing/tensor operations race on constrained hardware (Principle IV).
- *Cancel on navigation, as the earlier (pre-correction) draft spec assumed* — rejected per the corrected spec; would resurface the exact problem the correction called out.
- *Leave `lastSavedSession`-style single-pointer state and bolt a `conversationId` check on top without removing it* — rejected: this is precisely the "combining duplicated manually reconstructed history with hidden model-managed history" anti-pattern the task brief warned against; there must be one bounded ownership mechanism, not two overlapping ones.

**Test implication**: `tests/unit/store/inferenceStore.hydration.test.ts` already documents "no test of two threads being open/tracked at the same time" as a gap — this feature must add that coverage (`tests/integration/background-generation.test.ts` in plan.md's structure) as a failing test before the map-based ownership model is implemented, per Principle VI.

## R2. Per-turn image branching in the perception→answer pipeline (any turn, not only turn 1)

**Decision**: Generalize the "is this the first turn" check in `InferenceQueue.submit()`/`generateFirstImageAnswer()` to "does *this specific submitted turn* carry an attached image", independent of its position in the conversation. Concretely: `InferenceRequest` gains an optional `imagePath: string | null` (already effectively optional in spirit, now explicit) evaluated per call, not derived from whether a `lastSavedSession` exists. When the current turn has an image: run the existing two-stage pipeline unchanged (hidden structured-extraction call via `buildPerceptionModelMessages`, then the visible answer call via `buildAnswerPrompt`/`buildSingleUserModelMessages`) — but the answer-stage call additionally includes the conversation's bounded prior canonical turns (via `buildCanonicalModelMessages`-style context), not just the fresh extraction, so a later image-bearing turn stays properly connected to the conversation instead of the answer call always being a from-scratch single-user message. When the current turn is text-only: use the existing canonical-continuation path (`buildCanonicalModelMessages`) unchanged, which already lets it reference a prior turn's `pinnedExtraction`-derived visible content (formatted into the turn's stored answer text) without any special-casing.

**Rationale**: This directly implements FR-003/FR-009/FR-010/FR-011/**FR-033** — image attachment on any turn, later image turns don't reset context, text-only follow-ups can reference an earlier image-bearing turn's visible content without re-attachment, and *any ordering* of text/image turns (text→text, image→text, text→image→text, imageA→text→imageB→text, per FR-033's explicit list) — while preserving the two-stage perception→answer pipeline exactly as it exists today (per the task brief's explicit instruction), just triggered by a per-turn condition instead of a per-conversation one. Because the branch condition is purely "does this turn have an `imagePath`," no turn ordering is a special case: the same per-turn check handles a conversation with zero, one, or many image-bearing turns identically. This condition is now literally "does this `ConversationMessage` have a non-empty `attachments`" (data-model.md), evaluated per message rather than per array position; the existing double-context-injection guard — never feeding both the raw hidden extraction and its own already-derived visible text into the same prompt for one message — continues to apply unchanged, since it lives inside the frozen `ExtractionParser.ts`/`ContextBuilder.ts` logic this decision does not touch (research.md R9).

**Alternatives considered**:
- *Only ever allow one image per conversation, at any position (replace-the-slot model)* — rejected: contradicts FR-010 ("a later image-bearing message MUST NOT alter, remove, or reset any earlier message") and User Story 3's requirement that multiple image-bearing turns can coexist.
- *Re-run full extraction context for every subsequent turn regardless of whether it has a new image* — rejected: unnecessary perception-stage calls on text-only turns would double inference cost per turn and violate "keep hidden perception output internal-only and scoped to that conversation" in spirit (it should only run when there's actually new visual evidence to perceive).

**Verification required** (Principle IX): `research.md` does not need new library API verification here — this is pure application logic inside already-owned modules (`ContextBuilder.ts`, `InferenceQueue.ts`), no `react-native-executorch` surface changes.

## R3. Navigation: Conversation Drawer implementation

**Decision**: Add `@react-navigation/drawer` plus its required peer dependency `react-native-gesture-handler` (not currently installed), rather than hand-rolling a Reanimated-only drawer. `motion.md` §10 requires gesture-driven, low-overshoot-spring drawer settle behavior with a scrim that tracks drawer position — exactly what `@react-navigation/drawer` provides via native navigation transitions, satisfying `motion.md` §5's "native navigation transitions" and "gesture-handler-driven drawer interaction" guidance without reimplementing drawer physics.

**Rationale**: Constitution Principle V favors the simpler, already-solved approach over a custom implementation when one exists and fits; `react-native-reanimated`/`react-native-worklets` are already present but a maintained drawer package is lower long-term maintenance cost than a hand-built one.

**Alternatives considered**: Custom Reanimated-only drawer — rejected as higher implementation and maintenance cost for behavior an existing, actively maintained library already provides correctly.

**Verification required before adoption** (Principle IX + constitution's NDK development-workflow gate): Before installing, confirm (a) `@react-navigation/drawer`'s current major version is compatible with React Navigation 7 / RN New Architecture (Principle VII), and (b) neither `@react-navigation/drawer` nor `react-native-gesture-handler`'s `android/build.gradle` hard-requires NDK 27+ (the project is pinned to NDK 26.3.11579264 — check via `findstr /r /s "ndkVersion" node_modules\react-native-gesture-handler\android\*.gradle` per the existing AGENTS.md NDK rule). This check must happen during Phase 2 task execution, before the dependency is added, not assumed here.

## R4. Screen restructuring: one conversation-scoped screen, not a parallel chat system

**Decision**: Introduce a single screen (`ChatScreen.tsx`) that renders the New Chat empty state, Active Chat thread, Image Preview (unsent attachment), and Image Answer states as one component with internal state transitions — matching `design.md` §7.8/7.9/7.11/7.12/7.13, which all share one header/composer pattern. This **generalizes today's `AnswerScreen.tsx`** (renamed/restructured, not duplicated) rather than creating a second, parallel chat screen. `CaptureScreen.tsx`'s camera viewfinder is retained only as the capture mechanism reached via the composer's attachment control ("Attachment opens camera/gallery actions" per `screen_map.md`), not as the app's landing route. The landing route becomes `ChatScreen` in its New Chat state (FR-001/FR-013/SC-001).

**Rationale**: Directly satisfies the task brief's "do not create a parallel chat system" instruction and `screen_map.md`'s New Chat → Image Preview → Image Answer flow living inside one Chat destination, with Drawer/History as clearly separate navigation destinations.

**Alternatives considered**: Keep `CaptureScreen` as the landing route with a bolted-on "start text chat" affordance — rejected: contradicts SC-001 ("one continuous, unbroken flow... no separate or disconnected entry points") and `design.md` §6's `New Chat` being the primary destination, not the camera.

## R5. Component-level testing approach (no RNTL installed)

**Decision**: Continue the existing test convention. State/logic changes (conversation-keyed ownership, per-turn image branching, draft preservation, retry-in-place, title/preview/search derivation) get unit/contract tests against stores and pure functions. New screen wiring (`ChatScreen`, `ConversationDrawer`, rebuilt `HistoryScreen`) gets source-text structural assertions consistent with the existing `inferenceStore.hydration.test.ts` "screen wiring" pattern (regex/string assertions against literal screen source), not a new component-rendering dependency.

**Rationale**: No `@testing-library/react-native` is installed anywhere in the repo today; every existing test is pure-logic or structural. Introducing RNTL is a repo-wide testing-infrastructure decision with its own native-dependency/NDK-verification burden (Principle IX) that is out of proportion to a single feature, and Principle V favors not adding new abstractions/dependencies unless required.

**Alternatives considered**: Add RNTL now to properly render-test the new drawer/chat screens — rejected as scope creep for this feature; noted as a plausible future cross-cutting improvement, not a blocker here.

## R6. Scale validation (~200 conversations / ~200 messages) and virtualization

**Decision**: Use virtualized list rendering (`FlatList`) for both the conversation drawer/History list and the per-conversation message list — `design.md` §15 already calls for "virtualized lists for long chat/history lists where appropriate." The drawer's "recent conversations" subset uses `IHistoryStore.list(limit)` with a small limit (e.g., last ~20) since it only needs to show a handful of rows; Full History loads/paginates through the complete set via the same `list(limit, offset)` contract already implemented in `HistoryStore.ts`.

**Rationale**: Satisfies spec SC-003/SC-010's validation-dataset outcomes (no crash, no dropped/duplicated/reordered messages, usable navigation) without introducing a new pagination API or backend — everything is local MMKV, and `IHistoryStore.list()` already supports `limit`/`offset`. This scale target is explicitly a usability/validation bar per the corrected spec (not a hard product cap), so no new optimization infrastructure beyond standard `FlatList` virtualization is justified.

**Alternatives considered**: Build a dedicated pagination/indexing service ahead of need — rejected per the corrected spec's explicit instruction not to treat the ~200 figure as justification for unrelated optimization infrastructure.

## R7. Physical-device validation

**Decision**: `quickstart.md` includes a mandatory physical-device validation pass (Android device with 6GB+ RAM, per existing constitution Technology Constraints) covering: (a) creating and switching between multiple conversations without leakage, (b) a sustained multi-turn conversation (10+ turns, including at least one later image-bearing follow-up) for memory/performance behavior, (c) camera capture → attach → send on real camera hardware, (d) background generation — start a response, navigate to a different conversation and to the drawer/History, confirm the response completes and is correctly attributed when the user returns, and (e) drawer gesture responsiveness. Emulator results are declared non-authoritative for all of the above, consistent with existing project convention.

## R8. Title / preview / search derivation (deterministic, local-only)

**Decision**: Add pure functions (`deriveConversationTitle(conversation)`, `deriveConversationPreview(conversation)`, `searchConversations(conversations, query)`) in a new `src/history/ConversationSearch.ts` (title/preview derivation may live alongside or be re-exported from here; exact file boundary is a Phase 2 task-level decision), operating on `Conversation.messages: ConversationMessage[]` (data-model.md's canonical shape). Title = `messages[0].text` (always the conversation's first user message, per the Message Creation Invariant) if non-empty, else a fixed fallback string (e.g., "Image conversation") when that first message is image-only (non-empty `attachments`, empty `text`). Preview = the most recent user-visible message content — the last assistant message's `text` if its `status === 'completed'`, else the most recent user message's `text`. Search = case-insensitive substring match over `{title, and every message's text}` only — never `pinnedExtraction`, `hiddenEvidence`, or `InferenceTrace` fields, none of which are reachable from these functions' input shape in the first place (they only ever receive the already-visible `Conversation`/`ConversationMessage` fields, structurally enforcing FR-022/FR-024's "never from internal-only state"). These same functions back all four Full History recency groups (`Today`/`Yesterday`/`Previous 7 Days`/`Older`, design.md §7.14) — grouping is a display-time computation over `IHistoryStore.list()`'s results, not a separate derivation path (data-model.md "History grouping and pagination").

**Rationale**: Matches the existing codebase's precedent of deriving presentation data at read time from persisted visible fields rather than storing redundant derived state (e.g., `AnswerPostProcessor.assessAnswerQuality` is already re-run at `HistoryScreen` render time rather than persisted). Keeping these as pure functions with no access to internal-only fields makes "never leaks internal state into search/preview" a structural property, testable directly, rather than a runtime check that could be forgotten at a call site.

**Alternatives considered**: Persist a cached `title`/`preview` field on `Conversation`, recomputed on save — rejected as unnecessary state duplication for a cheap, pure, small-input computation; deferred unless Phase 2 profiling on the ~200-conversation validation dataset shows list-render cost from repeated derivation (unlikely given `FlatList` only computes visible rows).

## R9. Regression protection for the existing camera-first flow (FR-034/SC-014)

**Decision**: Treat `ExtractionPrompt.ts`, `AnswerPrompt.ts`, `SystemPrompt.ts`, `ExtractionParser.ts`, `AnswerPostProcessor.ts`, `GenerationLimits.ts`, and `GenerationTuning.ts` as **frozen for this feature** — none of them change. The only files that change to support any-turn images (R2) are the *trigger condition* in `ContextBuilder.ts`/`InferenceQueue.ts` (positional/turn-index check → per-message `attachments` check) and the *attribution* layer in `conversationStore.ts`/`inferenceStore.ts` (R1, now keyed by stable message identity rather than a turn index — see R1's update above). `tests/integration/vision-once-chat-flow.test.ts` — the existing test that locks in the turn-1-image camera-first flow end to end — is extended with any-turn/multi-image assertions but its existing turn-1 assertions are kept verbatim and must keep passing unmodified; a red run on any of its pre-existing assertions after R1/R2 land is treated as a regression, not an intentional behavior change to reconcile.

**Rationale**: FR-034 explicitly requires "no regression" and "generalizes when, not what" — the safest way to guarantee that in a task-decomposed implementation is to make the frozen-file list and the unmodified-assertions rule explicit and checkable, rather than trusting that a broad refactor happens to preserve behavior. This also directly satisfies the task brief's "protect the current known-good camera-first flow with regression coverage before generalizing" sequencing: the existing test file *is* that regression coverage, and R1/R2's tasks (Phase 2, Foundational, per tasks.md) are written test-first specifically so the old assertions fail loudly if disturbed.

**Alternatives considered**: Rewrite the prompt-construction/parsing modules alongside the ownership/branching changes, on the theory that "unifying" the pipeline should touch all of it — rejected per the task brief's explicit "do not scope Feature 003 as a rewrite of the existing inference-quality pipeline" and per Principle V (don't touch working code without a requirement forcing it).

**Verification required**: None beyond the existing test suite — this is a scoping/discipline decision for Phase 2 task execution, not a new library or API surface.

## R10. XState v5 as a narrow orchestration layer for the turn/inference lifecycle

**Codebase finding this responds to**: `InferenceQueue.ts` (`src/inference/InferenceQueue.ts`) is already, in effect, a hand-rolled state machine — a mutable `this.state: InferenceState` field with a `status: 'idle' | 'preprocessing' | 'loading_model' | 'streaming' | 'completed' | 'cancelled' | 'errored'` union, an `IN_FLIGHT` array checked imperatively (`isInFlight()`), and transitions expressed as a sequence of `this.setState({status: ...})` calls inside one large `try/catch/finally` in `submit()`. Two phases the corrected spec's UI needs to distinguish — image perception/extraction and prompt/context assembly — are *not* currently distinct `status` values at all: they happen silently inside the `'streaming'` status (see `generateFirstImageAnswer`'s extraction call and `generateFollowUpAnswer`'s `buildCanonicalModelMessages` call), which is exactly why `motion.md` §11.3's three real phases (`Preparing image…`, `Looking at image…`, `Thinking on-device…`) aren't cleanly implementable today without adding more ad hoc status strings to the same imperative pattern.

**Verification performed** (constitution Principle IX — this project's own memory records XState was never previously evaluated, so nothing here was assumed from training data): fetched current XState v5 documentation directly (`stately.ai/docs/xstate`, `stately.ai/docs/actors`) and searched for current package/React Native compatibility status as of this planning session. Confirmed: `setup({ actors, actions, guards }).createMachine({...})` is the current recommended construction API; `createActor(machine, { input }).start()` instantiates and runs it; `actor.subscribe(snapshot => ...)` (or an observer `{ next, error, complete }`) is the subscription API; `fromPromise(async ({ input }) => ...)` is for one-shot async steps (`invoke: { src, onDone, onError }`); `fromCallback(({ sendBack, receive, input }) => cleanupFn)` is for actor logic that must emit multiple events over time via `sendBack(...)` — the correct primitive for relaying streamed tokens, since `fromPromise` only ever resolves once. `xstate` itself is a zero-dependency pure-JS/TS library with documented React Native production usage; it requires no native module and therefore carries none of `research.md` R3's NDK-verification risk. `@xstate/react` (hooks for React components) is a *separate* package and is deliberately **not** added — see Decision below.

**Decision**: Introduce `xstate` (core package only, no `@xstate/react`) as the internal implementation of `InferenceQueue`'s state-transition logic — **`IInferenceQueue`'s existing public contract (`submit`, `cancel`, `subscribe`, `getState`) does not change**, and neither does anything outside `InferenceQueue.ts` that calls it (`conversationStore.ts`, `inferenceStore.ts` per R1, or any screen). Internally, `InferenceQueue` creates and holds one persistent machine actor (`createActor(turnLifecycleMachine).start()`, constructed once, mirroring the class's existing singleton lifetime) instead of hand-rolling `this.state`/`this.setState`. Each call to `submit()` sends the actor a `SUBMIT` event carrying the request (question, optional `imagePath`, a fresh `requestId`, and the `conversationId`/`originatingUserMessageId`/`assistantMessageId` attribution identities from R1, passed through untouched — the machine does not interpret them); `cancel()` sends `CANCEL`; `subscribe()` maps `actor.subscribe(...)` snapshots into the same `InferenceState` shape `IInferenceQueue.subscribe()` already emits today, so downstream code is unaffected.

States (mapping directly onto the task brief's requested vocabulary, and onto today's code):

| Machine state | Today's equivalent | What changes |
|---|---|---|
| `idle` | `status: 'idle'` | unchanged |
| `preparing` | `status: 'preprocessing'` | unchanged behavior (`prepareImageForInference`, only entered when the current turn has an image — R2); renamed to match the task brief's word and `motion.md`'s `Preparing image…` copy |
| `perception` | hidden inside `'streaming'` | **newly distinct** — wraps the existing, unmodified `buildPerceptionModelMessages`/`parseExtractionWithRetry` call as one `fromPromise` actor (the retry-once behavior stays *inside* that one actor, not re-modeled as machine sub-states — see Alternatives); only entered for an image-bearing turn |
| `contextAssembly` | hidden inside `'streaming'`/`'loading_model'` | **newly distinct**, brief/synchronous — wraps the existing, unmodified `buildCanonicalModelMessages`/`buildAnswerPrompt`/`buildSingleUserModelMessages` calls; always entered (both image-bearing and text-only turns) immediately before generation |
| `generating` | `status: 'loading_model'` | unchanged behavior (`engine.loadModel()` where still required); renamed for clarity — this is model preparation/dispatch, not yet token output |
| `streaming` | `status: 'streaming'` (token phase) | unchanged behavior; now implemented via a `fromCallback` actor that calls `sendBack({ type: 'TOKEN', response, count })` per token instead of the current raw `onToken` closure — this is the one place the *mechanism* changes, not the observable behavior (`InferenceState.response` still updates incrementally the same way) |
| `completed` | `status: 'completed'` | unchanged |
| `failed` | `status: 'errored'` | renamed to match spec FR-030's exact vocabulary |
| `interrupted` | `status: 'cancelled'` | renamed to match spec FR-030's exact vocabulary |

`RETRY` (FR-029) sent from `failed` re-enters the same per-turn flow (`preparing` if the turn's original request had an image, else straight to `contextAssembly`) reusing the **same** request payload already held in the machine's typed context (question, `imagePath`, `conversationId`, `originatingUserMessageId`, `assistantMessageId` — only `requestId` is refreshed) — never a new `originatingUserMessageId`/`assistantMessageId` — which is what makes "regenerate the same logical turn, no duplicate user message" a structural property of the machine rather than caller discipline.

**What the machine's XState `context` holds, and does not hold** (directly answering "do not use XState as conversation storage, context memory, or persistence"): only the single in-flight (or most recently completed/failed, until the next `SUBMIT`) turn's transient working data — the request being processed, the accumulating streamed response, the transient `hiddenEvidence`/`pinnedExtraction` for *that* turn, and the error message on failure. It is reset on every new `SUBMIT`. It never holds a conversation's `turns[]` array, is never the source of truth for what conversations exist, and is never read from or written to MMKV — `HistoryStore`/`conversationStore` remain exclusively responsible for all of that, unchanged from R1/data-model.md. (Note the deliberate terminology collision: the task brief's requested **`contextAssembly` state** — the step where an LLM prompt's `ModelRequestMessage[]` is built — is unrelated to XState's own generic term **`context`** — the machine's internal typed data bag. Both are used here because they're each the correct term in their own vocabulary; `contracts/turn-lifecycle-machine.md` keeps them clearly separated.)

**Rationale**: This satisfies every constraint in the task brief simultaneously: (a) "narrow orchestration layer" — it replaces exactly the control-flow logic `InferenceQueue.ts` already has, nothing more; (b) "only manage workflow states and transitions" — the listed states map 1:1 onto the task brief's own vocabulary; (c) preserves the Phase 002 pipeline, `ContextBuilder`, canonical ownership, and the visible/internal separation — every prompt-construction and parsing function (`ExtractionPrompt.ts`, `AnswerPrompt.ts`, `SystemPrompt.ts`, `ExtractionParser.ts`, `ContextBuilder.ts`) is called *from* an invoked actor, completely unmodified, per R9's frozen-file list; (d) Zustand remains the only UI-facing reactive layer — no component, hook, or store outside `InferenceQueue.ts` ever imports `xstate` or holds a machine reference; (e) app-wide single-flight is unchanged — there is exactly one machine actor instance, exactly as there is exactly one `InferenceQueue` instance today, and `InferenceActivityLock`/the `isInFlight()`-equivalent guard on `SUBMIT` while not `idle` is preserved as a machine guard.

**Alternatives considered**:
- *Do not adopt XState; keep the current hand-rolled status/if-else machine and just add two more status strings for perception/context-assembly* — rejected: this is exactly the kind of "combining duplicated, ad hoc state tracking" the earlier corrections in this spec's history explicitly warned against, and it's already showing strain (the perception/context-assembly phases had to be *inferred from code comments*, not read off `state.status`, during this session's research). XState formalizes a state machine that already exists, rather than introducing new complexity — this is why it clears Principle V despite being a new dependency.
- *Model perception's retry-once as explicit XState sub-states (`perception.extracting` → `perception.retrying` → `perception.done`)* — rejected for this feature: `ExtractionParser.ts`'s `parseExtractionWithRetry` already encapsulates that exact behavior correctly (per R9, it's frozen/unmodified); re-modeling it as machine sub-states would be restating working logic in a second place for no product-visible benefit, and risks subtly changing retry timing. One `perception` state invoking one `fromPromise` actor that calls the existing unmodified function is the narrower, lower-risk choice.
- *Add `@xstate/react` and let `ChatScreen`/`ChatComposer` consume the machine's snapshot directly via `useSelector`* — rejected: the task brief explicitly preserves "Zustand for reactive application/UI state"; introducing a second reactive-state library into components would violate that and duplicate what `conversationStore`'s existing `subscribeToConversation` (R1) already provides.
- *Spawn one machine actor per conversation instead of one persistent singleton* — rejected: would either violate app-wide single-flight (if actors could run concurrently) or require an extra coordination layer to enforce single-flight *across* actors, which is strictly more complexity than the current single-instance-with-a-lock approach for no behavioral gain, since only one turn is ever actually in flight at a time regardless.

## R11. Incremental migration sequencing (protect → introduce → verify → generalize → multi-conversation → UI → validate)

**Decision**: Adopt the task brief's 7-step sequencing as the mandatory ordering constraint on Phase 2 tasks, layered onto (not replacing) the Foundational/User-Story phase structure `tasks.md` already uses:

1. **Protect** — the existing `tests/integration/vision-once-chat-flow.test.ts` regression suite (R9) must be green *before* any XState code is written. This is a precondition check, not a new task.
2. **Introduce XState around the existing pipeline without changing observable behavior** — `InferenceQueue.ts`'s internals are reimplemented atop `turnLifecycleMachine` (R10) while its public `IInferenceQueue` contract stays byte-identical; this happens *before* R2's per-turn image-branching generalization, and *before* R1's conversation-keyed ownership map, so that if anything regresses, the cause is isolated to "the state-machine swap" and nothing else.
3. **Verify current image-first and follow-up behavior remains correct** — re-run `vision-once-chat-flow.test.ts` and `tests/contract/inference-pipeline.test.ts` unmodified against the XState-backed `InferenceQueue`; both must still pass with zero edits to their existing assertions (same regression gate as R9, now specifically re-checked immediately after step 2, not deferred to the end).
4. **Generalize the turn flow to support text-only and image-bearing turns at any point** — this is R2, now explicitly sequenced *after* the machine swap is verified stable, not concurrently with it.
5. **Support multiple isolated conversations, drafts, switching, History, and resume** — this is R1 (conversation-attributed ownership) plus the `HistoryStore`/`ConversationSearch` work from data-model.md, sequenced after step 4.
6. **Connect the redesigned UI to the unified turn workflow** — `ChatScreen`/`ChatComposer`/`ConversationDrawer`/`HistoryScreen` (R3, R4) are built last, against the by-then-stable store/inference layer — never against XState directly (R10's Zustand-only UI rule).
7. **Perform mixed multimodal and physical-device validation** — `quickstart.md` §2's full scenario set, run only once steps 1–6 are complete.

**Rationale**: This is the same "protect the known-good flow, then generalize" discipline R9 already established for the non-XState parts of this feature, extended to explicitly sequence the *new* architectural piece (XState) as its own isolated, independently-verified step rather than bundling it into the same change as the per-turn/multi-conversation generalization. Isolating "did the state-machine swap change behavior" from "did the any-turn-image generalization change behavior" as two separately-verified steps makes a regression's cause unambiguous if one appears.

**Alternatives considered**: Introduce XState and the per-turn image generalization in the same change, since they touch overlapping code (`InferenceQueue.ts`) anyway — rejected: this is exactly the "big bang" risk the task brief's incremental-migration instruction is designed to avoid; two independently-verifiable steps cost little extra sequencing discipline and materially reduce debugging risk if `vision-once-chat-flow.test.ts` goes red.

## R12. Navigation route consolidation: `Chat` as the single unified conversation destination

**Codebase finding this responds to**: `RootStackParamList` (`src/navigation/AppNavigator.tsx`) today defines `Answer: { imagePath: string; question: string; sessionId?: undefined } | { sessionId: string }` and `Capture: undefined`. Two existing call sites navigate to `Answer` directly: `src/screens/CaptureScreen.tsx:108` (`navigation.navigate('Answer', { imagePath: selectedImagePath, question: trimmedPrompt })`, immediately after `CaptureScreen` submits the question itself) and `src/screens/HistoryScreen.tsx:53` (`navigation.navigate('Answer', { sessionId })`, resuming a stored session). Left unaddressed, generalizing `AnswerScreen.tsx` into `ChatScreen.tsx` (R4) while these call sites still target `Answer` would leave two ambiguous, competing destinations.

**Decision**: `RootStackParamList.Answer` is renamed to `Chat`, with its params changed from the union above to a single required shape: `Chat: { conversationId: string }` (an existing conversation's id, or the sentinel `'new'` for an as-yet-uncreated New Chat draft, per data-model.md's `Draft.conversationId: string | null` — `'new'` is the route-param spelling of that `null` slot). `ChatScreen.tsx` (R4) is the only screen that ever renders this route. There is exactly one unified conversation destination in this feature; `Answer` does not exist afterward, either as a route name or as a lingering alias.

**Call sites migrated** (identified by direct inspection, not assumed — enumerated so none is missed during task execution):
- `src/navigation/AppNavigator.tsx` — `RootStackParamList.Answer` entry renamed to `Chat: { conversationId: string }`; `resolveInitialRoute()` points at `Chat` (with `conversationId: 'new'`) instead of `Capture` for a fresh launch, once onboarding/model setup is complete.
- `src/screens/CaptureScreen.tsx:108` — this call site's shape changes structurally, not just its route name: `CaptureScreen` no longer submits a question (that logic moves to `ChatComposer.tsx`, R13 below), so after a successful capture it calls `conversationStore.setDraftImage(conversationId, path)` and navigates back to `Chat` (or simply pops), rather than forward to a route carrying `imagePath`/`question`.
- `src/screens/HistoryScreen.tsx:53` — `navigation.navigate('Answer', { sessionId })` becomes `navigation.navigate('Chat', { conversationId: sessionId })`.
- `WelcomeScreen.tsx` — its "get started" action routes into `Chat` (`conversationId: 'new'`) once ready, instead of `Capture`.
- `ConversationDrawer.tsx` (R3) — both resuming an existing conversation and the "start new conversation" action navigate to `Chat`, with an existing id or `'new'` respectively — never to `Answer`.

**Rationale**: The task brief is explicit that `Answer` and `Chat` must not exist as two ambiguous, competing destinations, and that migration of every existing call site must be planned explicitly rather than left implicit. A rename (not an addition alongside a lingering `Answer`) is the only choice that satisfies this; enumerating every current call site above (found by direct code inspection) closes the gap flagged by this feature's `/speckit-analyze` pass, where the route-rename decision and its call-site migration were implied but never stated.

**Alternatives considered**: Keep the `Answer` route name and only widen its params to `{ conversationId }` — rejected: the name itself is developer-visible technical debt that keeps inviting the question "is `Answer` still the real destination, or is `Chat`," which is exactly the ambiguity the task brief asks to close.

## R13. Capture and attachment source-selection ownership (Camera/Gallery, any conversation, any turn)

**Codebase finding this responds to**: today, `CaptureScreen.tsx` both picks/captures an image *and* submits a question directly via `useInferenceStore`'s `submit`, with no concept of "which conversation this belongs to" beyond there being exactly one active thread app-wide. Once multiple conversations can exist simultaneously (R1) and a New Chat draft can exist before any conversation is created (data-model.md `Draft`), an attachment flow that doesn't know its own owner risks attaching an image to whichever conversation happens to be active when the picker/camera returns — precisely the FR-021 leak category this feature must prevent.

**Decision**: The design source (`design/design.md`, "Attachment Source Selection", updated this planning pass) now defines the UI: tapping the composer's image/media action presents `Camera` / `Gallery` / `Cancel`. The technical mechanism for returning a result to the correct owner:
- **Gallery**: handled entirely within the composer's own screen, no navigation — `useMediaStore`'s `pickImageFromLibrary()` is awaited inline, and the result is written directly into *that screen's own* `conversationId`'s draft via `conversationStore.setDraftImage(conversationId, path)`. Because no navigation occurs, there is no window during which "the active conversation" could change out from under the picker call.
- **Camera**: requires navigating to `CaptureScreen` (R12's renamed-route discipline applies here too), so `CaptureScreen`'s route params carry a required `conversationId: string` (an existing id, or `'new'` for a not-yet-created draft — same sentinel as R12). On a successful capture, `CaptureScreen` calls `conversationStore.setDraftImage(conversationId, path)` using that route-param identity — never an ambient "current conversation" — and navigates back. This holds identically whether `conversationId` is an existing conversation (a later-turn attachment) or `'new'` (a first-turn attachment, before any conversation exists yet).
- `CaptureScreen.tsx` is stripped of its own prompt input and its own `submit()` call — both move to `ChatComposer.tsx` — so `CaptureScreen`'s only remaining job is "capture a photo and hand its path back to the owner that opened it."

This covers, by construction rather than by convention: first-turn attachment (`conversationId: 'new'`), later-turn attachment (`conversationId: <existing id>`), conversation switching while the picker/camera is open (irrelevant — the owner identity travelled in the route param or the calling closure, never from ambient "current" state), and New Chat draft attachment (the `'new'` slot, per data-model.md `Draft`).

**Rationale**: Directly satisfies the requirement that Camera/Gallery results return to "the initiating draft owner rather than being attached using whichever conversation happens to be active when the result returns." Passing `conversationId` as an explicit route param (Camera) or capturing it in the calling closure (Gallery, no navigation) are the two mechanisms this codebase's existing React Navigation + Zustand architecture already provides; neither requires a new cross-cutting "current conversation" concept, which is exactly the ambient-pointer anti-pattern R1 already rejects for inference ownership.

**Alternatives considered**: A single app-wide "currently active conversation for attachment purposes" pointer, updated on navigation focus — rejected: this reintroduces the same ambient-pointer anti-pattern R1 replaces for inference ownership, and would recreate exactly the race this decision exists to prevent (the picker/camera returning after the user has already switched conversations).
