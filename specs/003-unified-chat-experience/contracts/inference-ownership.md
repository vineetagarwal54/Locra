# Contract: Inference Ownership & Background Generation

Documents the behavioral contract between `conversationStore`, `InferenceQueue`, and the UI layer for FR-012 through FR-016 and FR-021 (app-wide single-flight with conversation-attributed background continuation). This is the most safety-critical contract in this feature — it directly implements the corrected spec's central architecture change (research.md R1).

**Canonical shape note**: ownership is keyed by `conversationId` plus the stable `originatingUserMessageId`/`assistantMessageId` pair (data-model.md), never by a positional turn index. Wherever this document says "the owning message" it means the specific `ConversationMessage` identified by `assistantMessageId`.

## Ownership attribution

- Exactly one conversation may be the "active generation owner" at any instant, or none (idle). Ownership is assigned the instant `submit()`/`retryFailedMessage()` successfully acquires the `InferenceActivityLock`/starts the `InferenceQueue`, and released the instant that generation reaches a terminal state (`completed`, `failed`, or `interrupted`).
- Ownership is **not** tied to which screen is currently mounted/focused. A conversation retains ownership while the user navigates to a different conversation, opens the drawer, or opens History.
- **Navigation must never redirect streaming output based only on the currently visible conversation.** Streamed tokens are written **only** into the owning conversation's `ConversationRuntimeState.streamingText` / eventual `ConversationMessage` (identified by `conversationId` + `assistantMessageId`, set once at submit time and never re-derived from screen focus). No other conversation's state is ever touched by a given generation's callbacks, and switching which conversation is on screen never changes which conversation/message a running generation writes into.
- The `ModelRequestMessage[]` context assembled for the in-flight message (via `ContextBuilder.ts`, research.md R2) is likewise scoped to the owning conversation only — it is built from that conversation's own bounded prior messages and is never read by, or leaked into, any other conversation's context assembly (FR-021's "model request context" leak category).
- Every active inference remains explicitly associated with four identities for its entire lifetime: a `requestId` (this specific submit/retry attempt), the `conversationId` it belongs to, the `originatingUserMessageId` it is answering, and the `assistantMessageId` it is writing into (data-model.md `TurnLifecycleSnapshot.context.request`). All four are fixed at submit time and unchanged by navigation, screen remounts, or a subsequent retry (retry reuses the same `originatingUserMessageId`/`assistantMessageId`, minting only a fresh `requestId`).

## UI contract (what every conversation-scoped screen must do)

1. On mount/focus, a screen for `conversationId` subscribes to `conversationStore.subscribeToConversation(conversationId, ...)` and renders based on that conversation's own `ConversationRuntimeState` — never a global/shared "current inference state."
2. The composer for `conversationId` MUST be in a locked/disabled state whenever `conversationStore.isAnyGenerationInFlight()` is true **and** `getActiveGenerationOwner() !== conversationId` — i.e., "someone else is generating" is a distinct, indicated condition, not silently identical to "this conversation is generating" (FR-016). The two locked states MAY share the same visual "locked" composer treatment already defined in `motion.md` §12 ("Locked during inference"), but the underlying reason (own vs. elsewhere) must be programmatically distinguishable for test purposes even if the visual treatment is similar.
3. A cancel/stop control is rendered only when `getActiveGenerationOwner() === conversationId` for the screen currently showing that conversation — there is no cross-conversation cancel affordance (per spec Edge Cases: "Cancellation is only available from within the generating conversation's own composer/stop control").
4. On unmount, a screen MUST NOT call `cancelActiveGeneration()` for its conversation (this is the behavior that changes from today's `AnswerScreen`) — leaving the screen must never stop the generation.
5. A screen never derives "which message is being generated" from array position (e.g., "the last message in the list"); it reads `ConversationRuntimeState.assistantMessageId` and matches it against the specific `ConversationMessage.id` it is rendering. This is what keeps rendering correct even if a screen is mid-transition or the list is virtualized and not fully mounted.

## Test scenarios this contract must be verified against (Phase 2 — write failing tests first, per Principle VI)

1. Start a generation in conversation A; while it streams, switch to conversation B; assert B's runtime state shows no streaming content and A's continues to update, attributed to A's specific `assistantMessageId`.
2. While A is generating, attempt `submit()` for conversation B; assert it is rejected/no-op, B's draft is unchanged (preserved, not cleared), and A's generation is unaffected.
3. Let A's generation complete while conversation B is on screen; assert A's `HistoryStore` entry reflects the completed message (matched by `assistantMessageId`, not by position); assert no visible indicator appears in B (no unread badge, per the design guardrail) and no error is raised.
4. From conversation B, open the drawer and select A while A is still generating; assert A's screen renders the live in-progress `generating` state for the correct `assistantMessageId` (not `interrupted`, not stale, not mismatched to the wrong message).
5. From within A while it is generating, tap cancel; assert the pending message (identified by `assistantMessageId`) transitions to `interrupted`, `InferenceActivityLock`/`InferenceQueue` return to idle, and a subsequent `submit()` from any conversation succeeds.
6. Rapid repeated `submit()` calls for the same conversation A while A is already generating; assert only one generation is ever created (existing `IN_FLIGHT` rejection behavior, re-verified conversation-scoped) and no second `assistantMessageId` is minted for the same user message.
